## 0. 先看结论

当前目标“探索 Cloudflare Business Plan 最佳安全配置和性能配置”已经完成方案收敛；现阶段是**实施准备 / 只读核查阶段**，不是生产变更阶段。

可以直接实施的首批 P0 措施主要在 Cloudflare 控制台完成：

1. 核查 `0g.hk`、`*.0g.hk` DNS / Route / Proxy 是否都经过 Cloudflare 并命中 Worker；
2. 收敛 TLS / HTTPS / HSTS 基线；
3. 启用 WAF Managed Rules，但首轮建议先观察 / Log / Simulate，不要一上来全量 Block；
4. 给 `/exists` 加边缘 Rate Limiting，防止枚举名称；
5. 给创建、编辑、举报等写路径加边缘 Rate Limiting；
6. 给 `/admin` 与 `/admin/*` 加 Cloudflare Access；
7. 检查 Cache Rules，确保 admin、写路径、编辑页、`/exists` 不被误缓存。

暂时不要做的事：

- 不要开启全站 Under Attack / 全站 Challenge；
- 不要给所有用户强制 Turnstile；
- 不要设置激进的 Cache Everything；
- 不要在没有处理 Bearer/API 调用兼容性的情况下，一刀切阻断所有 `/admin*` 脚本访问；
- 不要把本文的“建议参数”当成已经核实现网状态。实际控制台状态仍需只读核查。

当前剩余关键依赖：

- 只读 Cloudflare 访问审批已显示 approved，但仍需确认实际访问材料可用，并完成现网差异核查；
- 产品边界仍需最终确认：`GET /?c=` 是否移除/收紧、`/admin*` Access 对 Bearer/API 的影响、`curl/API` 主路径兼容性；
- 任何真正修改生产控制台配置或 Worker/KV 部署，都应单独走 production_deploy / Board / CEO 审批。

## 3. 可直接实施的 P0 控制台措施

下面是你可以自己在 Cloudflare 控制台执行或核查的措施。建议严格按顺序做。

---

### P0-1：核查 DNS / Route / Proxy

**目标**：确保 `0g.hk` 与 `*.0g.hk` 都经过 Cloudflare，并正确命中 Worker。

**控制台位置**：

- Cloudflare → Websites → `0g.hk` → DNS → Records；
- Cloudflare → Workers & Pages → 对应 Worker → Settings / Triggers / Custom Domains / Routes。

**核查项**：

| 项 | 期望 |
|---|---|
| `0g.hk` | 绑定到当前 Worker 或 custom domain 正常 |
| `*.0g.hk/*` | wildcard route 生效 |
| DNS proxy | 承载业务的记录应为 Proxied / 橙云 |
| 历史 DNS-only 记录 | 若会绕过 Cloudflare，应清理或确认无业务风险 |

**验证命令**：

```bash
curl -I https://0g.hk/
curl -I https://test-random-name.0g.hk/
dig 0g.hk
dig test-random-name.0g.hk
```

**通过标准**：

- `https://0g.hk/` 正常返回；
- 随机子域请求经过 Cloudflare；
- 没有业务 wildcard 绕过 Cloudflare 代理。

**回滚方式**：

- 恢复变更前 DNS / route / custom domain 截图中的状态；
- 如新增 route 出错，立即禁用或删除新增 route。

---

### P0-2：TLS / HTTPS / HSTS 基线

**目标**：固定 HTTPS 与 TLS 安全基线，但避免 HSTS preload 这种难回滚配置。

**控制台位置**：

Cloudflare → Websites → `0g.hk` → SSL/TLS。

**建议配置**：

| 项 | 建议 |
|---|---|
| Always Use HTTPS | On |
| Minimum TLS Version | TLS 1.2 或更高 |
| TLS 1.3 | On |
| HTTP/3 | On |
| HSTS | 可开启，但首轮用短 max-age，例如 86400；不要 preload |
| HSTS includeSubDomains | 只有确认所有子域都必须 HTTPS 时再启用 |

**验证命令**：

```bash
curl -I http://0g.hk/
curl -I https://0g.hk/
```

**通过标准**：

- HTTP 自动跳转到 HTTPS；
- HTTPS 正常；
- 如启用 HSTS，响应头有 `strict-transport-security`；
- 不出现证书错误。

**回滚方式**：

- 关闭 Always Use HTTPS / HSTS 或恢复旧值；
- 若已设置长 HSTS，浏览器端回滚会有滞后，所以首轮不要设置长 max-age，更不要 preload。

---

### P0-3：WAF Managed Rules 观察模式

**目标**：先让 Cloudflare 记录通用攻击、扫描、异常请求，确认误伤后再逐步提升动作。

**控制台位置**：

Cloudflare → Security → WAF → Managed rules。

**建议动作**：

| 项 | 建议 |
|---|---|
| Cloudflare Managed Ruleset | Enable |
| OWASP Core Ruleset（如可用） | Enable |
| 初始动作 | Log / Simulate / Observe 优先 |
| 观察时间 | 24–48 小时 |
| 重点路径 | `/admin*`、`/exists`、`POST/PUT /`、`*.0g.hk` 写路径、`/raw` |

**为什么不要一上来 Block 全部**：

0g.hk 允许用户提交文本、Markdown、代码片段和 URL。正常内容可能像 SQL、XSS、命令行或安全 payload。Managed Rules 直接全量 Block 可能误伤合法笔记。

**验证方式**：

- 打开 Security Events / WAF Analytics；
- 过滤 `host=0g.hk`、`path=/admin`、`path=/exists`、`method=POST/PUT`；
- 查看 rule id、Ray ID、action、sample path。

**提升动作建议**：

| 类别 | 初始 | 观察后 |
|---|---|---|
| Protocol anomaly / malformed request | Log | Block |
| 已知 CVE probing | Log | Block |
| `/admin*` 高置信攻击 | Log | Block 或 Managed Challenge + Access |
| SQLi/XSS body signatures | Log | 谨慎，逐条确认，避免误伤代码笔记 |
| `/raw` 命中 | Log | 通常保持 Log，避免因内容文本阻断读取 |

**回滚方式**：

- 将对应 Managed Rules override 恢复为 Log / Simulate；
- 禁用导致误伤的单条 rule id，不要整站粗暴关闭所有安全能力，除非出现大面积故障。

---

### P0-4：`/exists` 边缘限流

**目标**：防止脚本枚举可用名称，减少 Worker/KV 压力。

**控制台位置**：

Cloudflare → Security → WAF → Rate limiting rules。

**建议规则**：

| 字段 | 推荐值 |
|---|---|
| 规则名 | `0g Phase2 RL-EXISTS-P0` |
| 表达式 | `http.host eq "0g.hk" and http.request.uri.path eq "/exists" and http.request.method eq "GET"` |
| 计数维度 | IP |
| 阈值 | 60 requests / 60 seconds |
| 动作 | 429 或 Managed Challenge |
| Mitigation timeout | 10 minutes |

**动作选择建议**：

- 如果担心 API / 自动化调用收到人机验证页，选 429；
- 如果确认只有浏览器使用并能接受挑战，可选 Managed Challenge；
- 首轮不要低于 60/min，避免误伤首页输入框 debounce。

**验证方式**：

```bash
curl 'https://0g.hk/exists?n=phase2-test'
```

超阈值测试只能在获批窗口做，不要随意打生产流量。

**通过标准**：

- 单次正常查询返回 JSON；
- 高频枚举在 Security Events / Rate Limiting Analytics 中可见；
- 首页输入校验不受影响。

**回滚方式**：

- Disable 该 Rate Limiting rule；
- 或把阈值提高到 120/min；
- 或把动作从 Challenge 改为 429 / Log。

---

### P0-5：创建 / 编辑 / 举报写路径限流

**目标**：把匿名创建、编辑、举报等高风险写操作先在边缘限速。

**控制台位置**：

Cloudflare → Security → WAF → Rate limiting rules。

#### 规则 A：主域创建 API

| 字段 | 推荐值 |
|---|---|
| 规则名 | `0g Phase2 RL-CREATE-P0` |
| 表达式 | `http.host eq "0g.hk" and http.request.uri.path eq "/" and (http.request.method eq "POST" or http.request.method eq "PUT")` |
| 阈值 | 10 requests / 60 seconds / IP |
| 动作 | 429 / Block，优先 429 |
| Timeout | 60–120 seconds |

#### 规则 B：子域编辑 / 写操作

| 字段 | 推荐值 |
|---|---|
| 规则名 | `0g Phase2 RL-SUB-WRITE-P0` |
| 表达式 | `ends_with(http.host, ".0g.hk") and (http.request.method eq "POST" or http.request.method eq "PUT")` |
| 阈值 | 10 requests / 60 seconds / IP |
| 动作 | 429 / Block，优先 429 |
| Timeout | 60–120 seconds |

#### 规则 C：举报接口

| 字段 | 推荐值 |
|---|---|
| 规则名 | `0g Phase2 RL-ABUSE-P0` |
| 表达式 | `ends_with(http.host, ".0g.hk") and http.request.uri.path eq "/abuse/report" and http.request.method eq "POST"` |
| 阈值 | 5 requests / 60 seconds / IP |
| 动作 | 429 或 Managed Challenge |
| Timeout | 10 minutes |

**为什么举报阈值更低**：

当前举报 3 次可能触发自动禁用。举报接口如果被批量滥用，会造成误封，所以外层阈值应更保守。

**验证方式**：

- 正常创建 1–2 条测试内容；
- 正常编辑测试内容；
- 正常提交一次举报；
- 在 Security Events 里确认规则可见；
- 超阈值测试只在审批窗口执行。

**回滚方式**：

- 禁用对应单条规则；
- 或提高阈值；
- 如果 API 收到 HTML challenge，立即把动作改为 429，不要让 CLI/API 卡在人机验证页。

---

### P0-6：`/admin` 与 `/admin/*` 启用 Cloudflare Access

**目标**：让公网访客不能直接看到应用层 admin 登录页，把管理面从公开互联网隔离到可信身份之后。

**控制台位置**：

Cloudflare Zero Trust → Access → Applications → Add application → Self-hosted。

**建议配置**：

| 字段 | 推荐值 |
|---|---|
| Application type | Self-hosted application |
| Application domain | `0g.hk` |
| Path | `/admin` 和 `/admin/*`，如 UI 只能通配再用 `/admin*` |
| Session duration | 4 小时起步，最多 8 小时 |
| Policy | Allow 指定管理员，默认 Deny all |
| Include | CEO / CTO / Security / Ops 指定邮箱或 IdP group |
| Require | MFA / OTP / GitHub org 等强身份条件 |
| Service Auth | 若脚本 API 仍需访问 `/admin*`，必须创建 Access Service Token |

**重要边界**：

当前 `/admin` 可能同时承载 Web UI 和 Bearer-auth JSON API。若直接给 `/admin*` 加 Access，脚本 API 可能被挡。实施前必须二选一：

1. 给脚本加 Access Service Token，并继续带原有 `Authorization: Bearer ...`；
2. 后续把脚本 API 从 `/admin*` 拆到独立路径；
3. 不推荐为了脚本 API 让 `/admin*` 对公网开放。

**验证方式**：

| 验证项 | 通过标准 |
|---|---|
| 隐身窗口访问 `https://0g.hk/admin` | 先看到 Access 身份验证或拒绝，而不是应用 admin 登录页 |
| 管理员登录 Access 后访问 | 能进入应用层 admin 登录 / dashboard |
| 非管理员访问 | Access Denied，Access Logs 有记录 |
| Service Token API（如保留） | 带 `CF-Access-Client-Id` / `CF-Access-Client-Secret` / `Authorization: Bearer` 可访问；缺任一材料失败 |

**回滚方式**：

- Disable Access application；
- 或临时放宽策略给管理员组；
- 若脚本 API 被阻断且无替代路径，立即回滚 Access 或启用 Service Token。

---

### P0-7：Cache Rules 防误缓存

**目标**：防止动态路径、写路径、admin、编辑页被错误缓存，同时保留公共读取路径的短缓存语义。

**控制台位置**：

Cloudflare → Rules → Cache Rules。

**必须 Bypass / Respect Origin 的路径**：

| 路径/条件 | 建议 |
|---|---|
| `/admin*` | Bypass Cache |
| `/exists` | Bypass Cache |
| `*/edit` | Bypass Cache |
| 所有 POST/PUT | Bypass Cache |
| 带 `Authorization` 或 edit token 的请求 | Bypass Cache |
| 302 / 中间页 / JSON API | Respect Origin，避免 Cache Everything |

**可短缓存的路径**：

| 路径 | 建议 |
|---|---|
| `/raw` | 尊重 Worker 当前短缓存，例如 60s；不要长期缓存 |
| `/llms.txt`、说明性静态内容 | 可设置更稳定 TTL，但需验证内容一致 |

**验证命令**：

```bash
curl -I https://0g.hk/
curl -I https://0g.hk/llms.txt
curl -I https://example-sub.0g.hk/raw
curl -I https://0g.hk/exists?n=phase2-test
```

观察：`cache-control`、`cf-cache-status`。

**通过标准**：

- admin / exists / edit / write 不被缓存；
- `/raw` 不出现长期旧内容；
- 创建和编辑后的内容在预期窗口内刷新；
- 不缓存 `X-Edit-Token`、`X-Edit-Url`、JSON 创建响应。

**回滚方式**：

- Disable 新增 Cache Rule；
- Purge 相关缓存；
- 恢复到 Respect Origin。

## 4. P1：建议实施但需要工程/产品确认的事项

这些不建议你直接在控制台单独做，因为会改变产品行为或 Worker 运行配置。

### P1-1：收紧或移除 `GET /?c=` 一键创建

当前 `GET /?c=...` 会触发创建，是有副作用的 GET。风险是第三方页面、聊天工具、邮件预览可能诱导浏览器发起创建。

建议：

- 最优：工程移除该入口，创建只走 POST；
- 若暂时保留：限制为浏览器 UI、叠加更严格 Rate Limit 或 Turnstile；
- 建议临时规则：`GET /?c=` 每 IP 3 req / 60s。

需要产品确认是否保留这个体验。

### P1-2：`/admin*` Bearer/API 路径收敛

建议：

- 移除 `?key=` URL query 鉴权；
- 保留 Bearer；
- 若 Access 保护 `/admin*`，脚本 API 需增加 Access Service Token；
- 中长期把 Web admin 与 API 路径拆开。

### P1-3：Worker runtime secrets

这些不是普通 zone 配置，属于 Worker 运行时配置：

- `ADMIN_KEY`；
- `ADMIN_SESSION_SECRET`；
- `SAFE_BROWSING_KEY`；
- 如接 Turnstile，再加 `TURNSTILE_SECRET`。

需要通过 Worker dashboard 或 `wrangler secret put` 设置，并走部署/变更审批。

### P1-4：Managed Rules 从观察提升到 Block

24–48 小时观察后，可以把低误伤、高置信规则提升为 Block：

- malformed request；
- known exploit / CVE probing；
- 与产品正文无关的协议异常；
- `/admin*` 高置信攻击。

对 SQLi/XSS body signatures 要谨慎，因为正常笔记可能包含代码片段。

## 5. P2：条件触发，不作为首轮默认动作

| 项 | 触发条件 | 注意事项 |
|---|---|---|
| Turnstile | 浏览器创建/举报 abuse 明显，且 Rate Limit 不够 | 只覆盖浏览器流量，不要破坏 curl/API |
| `/exists` 应用内审计/限流 | 边缘限流仍不足，需业务日志 | 需要改 Worker/KV |
| fail-open 改 fail-closed | 安全要求高于可用性，且误伤可接受 | AI/Safe Browsing 故障可能影响创建成功率 |
| allowlist / brand / TTL 调整 | 出现明确 abuse campaign 或产品策略变化 | 属于产品行为变更，不应只在控制台漂移 |
| 自动告警 | 规则稳定后需要运维值守 | 避免告警噪声过大 |


|---|---|

### 6.3 业务验收

| 主路径 | 必须保持 |
|---|---|
| 首页浏览 | 正常 |
| 浏览器创建 | 正常，除非产品决定改造 |
| curl/API 创建 | 不被 Challenge 页面破坏 |
| 子域读取 | 正常 |
| `/raw` | 正常，短缓存不展示长期旧内容 |
| 编辑 | 正常，edit token 不被缓存泄漏 |
| 举报 | 正常低频可用，批量被限速 |
| admin | 管理员可用，非管理员被 Access 拦截 |

---

## 7. 回滚规则

出现以下任一情况，立即停止后续批次并回滚相关变更：

1. 关键路径 5xx 明显升高；
2. 创建/编辑成功率明显下降；
3. 合法 admin 或 API 被阻断且没有可用替代路径；
4. admin、写路径、JSON、edit token 被缓存；
5. Security Events 显示大量合法请求被误伤；
6. 用户反馈集中出现无法创建、无法编辑、无法访问；
7. CEO/CTO/安全任一负责人触发 no-go。

回滚优先级：业务主路径可用性 > 管理面可恢复 > 保留安全规则 > 完成窗口范围。

---

## 8. 推荐实施顺序（最简执行版）

如果你只想照着做，按这个顺序：

1. **只读核查**：截图/记录 DNS、route、TLS、WAF、Rate Limit、Access、Cache 当前状态；
2. **TLS/HTTPS**：开启或确认 Always HTTPS、TLS 1.2+、TLS 1.3、HTTP/3；HSTS 先短 max-age，不 preload；
3. **WAF Managed Rules**：启用观察模式 24–48h；
4. **Rate Limit `/exists`**：60/min/IP，动作 429 或 Managed Challenge；
5. **Rate Limit 创建/编辑/举报**：创建 10/min，子域写 10/min，举报 5/min；优先 429；
6. **Access `/admin` + `/admin/*`**：先确认管理员身份与脚本 API；必要时配 Service Token；
7. **Cache Rules**：明确绕过 admin、exists、edit、POST/PUT；保留 `/raw` 短缓存；
8. **观察 24h**：看 Security Events、4xx/5xx、创建/编辑成功率、Access Logs、Cache；
9. **观察 7d**：再决定是否把部分 WAF rule 从 Log 提升到 Block；
10. **再做 P1/P2**：处理 `GET /?c=`、admin API、runtime secrets、Turnstile 等工程/产品项。

---

## 9. 最小变更审批包建议

如果要进入实际生产配置变更，建议审批包只包含 P0，且分批执行：

### 批次 1：只读核查

- 不改配置；
- 输出现网差异；
- 风险最低。

### 批次 2：TLS + WAF 观察

- TLS/HTTPS 小幅收敛；
- WAF 只观察，不全量阻断；
- 可快速回滚。

### 批次 3：Rate Limiting

- `/exists`；
- 创建/编辑/举报；
- 先宽阈值，优先 429。

### 批次 4：Access admin

- 仅在管理员身份、Service Token/API 边界确认后做；
- 最容易误伤脚本 API，所以单独批次。

### 批次 5：Cache Rules

- 重点防误缓存；
- 变更后立即验证创建/编辑/raw/admin。

## 10. 一页决策表

| 措施 | 能否直接做 | 优先级 | 主要风险 | 回滚难度 |
|---|---|---:|---|---:|
| DNS/route/proxy 只读核查 | 可以 | P0 | 无变更风险 | 低 |
| Always HTTPS / TLS 1.2+ / TLS 1.3 / HTTP3 | 可以，建议审批后 | P0 | 个别老客户端兼容 | 低 |
| HSTS 短 max-age | 可以，谨慎 | P0/P2 | 长 HSTS 难回滚 | 中 |
| HSTS preload | 不建议首轮 | P2 | 极难回滚 | 高 |
| WAF Managed Rules 观察 | 可以 | P0 | 低 | 低 |
| WAF 全量 Block | 不建议直接做 | P1 | 误伤文本/代码内容 | 中 |
| `/exists` Rate Limit | 可以 | P0 | 输入校验误伤 | 低 |
| 写路径 Rate Limit | 可以 | P0 | API 被 Challenge 破坏 | 低-中 |
| `/admin*` Access | 可以，但需先处理 API | P0 | 脚本 API / 管理员被锁 | 中 |
| Cache Rules Bypass 动态路径 | 可以 | P0 | 配错导致缓存污染 | 中 |
| Cache Everything | 不建议 | 禁止首轮 | 动态内容/Token 泄漏 | 高 |
| 移除 `GET /?c=` | 需工程/产品 | P1 | 产品体验变化 | 中 |
| Turnstile 全量 | 不建议 | P2 | curl/API 失效 | 中-高 |
| Turnstile 仅浏览器创建 | 需工程 | P2 | 转化下降 | 中 |

## 11. 最终 CEO 建议

我建议按“保守、可验证、可回滚”的路径推进：

1. 先完成只读核查，确认现网差异；
2. 第一批只做低风险 P0 控制台项：TLS/HTTPS、WAF 观察、Rate Limit、Cache Bypass；
3. `/admin*` Access 单独作为一批，先确认 Bearer/API 和 Service Token；
4. 任何 P1/P2 工程项不要混入第一批控制台变更；
5. 每项变更都要有截图、验证、回滚负责人和 24h/7d 观察记录。

这样做的好处是：安全收益最大、对现有 `curl/API` 和匿名创建产品定位冲击最小、出现误伤也能按单条规则快速回滚。
